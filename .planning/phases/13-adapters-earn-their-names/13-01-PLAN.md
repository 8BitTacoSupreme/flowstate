<objective>
Give the research adapter Autoresearch's core mechanism: measure -> keep/discard applied to OUTPUT. After a topic section is generated, score it for groundedness against the active fixture's `retrieval_questions`; retry a weak section within a bounded budget, then discard it if still weak. Record kept vs discarded.

Purpose: MECH-01 — the `research` adapter currently only fan-outs and concatenates. It must now actually judge its own output and drop ungrounded sections, without ever modifying the prompt (prompt tuning stays in `bench/`, per locked decision).

    - A section that scores below threshold is regenerated (same prompt, unmodified) up to a bounded groundedness-retry budget; if it clears the threshold on a retry it is KEPT.
    - A section that never clears the threshold within the budget is DISCARDED (excluded from the report body) and recorded as discarded.
    - When the fixture is absent or has empty `retrieval_questions`, scoring is SKIPPED and all bridge-successful sections are kept (graceful degrade, recorded as "scoring skipped: no fixture").

    Add a module helper `_load_retrieval_questions(root: Path) -> list[str]` that reads `.planning/fixtures/starter.json` (reuse the `_FIXTURE_PATH` string value `.planning/fixtures/starter.json`), `json.loads` it inside a try/except returning `[]` on any failure, and returns the `retrieval_questions` list (or `[]`).

    Add a method `_score_groundedness(self, section: str, questions: list[str]) -> float` that issues ONE `self.bridge.run(...)` scoring call. The scoring prompt asks the model to rate, on a 0-10 integer scale, how well the given research SECTION is grounded in / addresses the listed `retrieval_questions`, and to output ONLY the integer. Parse the returned score with a bounded parser: extract the first integer via a strict regex (`re.search(r"-?\d{1,3}", output)`), clamp to 0-10, and return it as a float normalized to 0.0-1.0 (score/10). NEVER use `eval`/`exec`/`ast.literal_eval` on model output. On bridge failure or unparseable output, return `0.0` (treated as weak). Use `model="sonnet"`, a low `max_turns` (e.g. 2), and NO tools (`allowed_tools=[]`) so the judge cannot browse.

    Add module constants `_GROUNDEDNESS_THRESHOLD = 0.6` and `_GROUNDEDNESS_MAX_RETRIES = 1` (bounded budget: at most one regeneration beyond the initial success per section).

    In `execute()`, after a topic's bridge call succeeds (`br.success and br.output.strip()`), when `questions` is non-empty, call `_score_groundedness`. If the score >= threshold, keep the section. If below threshold and the groundedness-retry budget for this topic is not exhausted, regenerate the section with the SAME `prompt` (do not alter the prompt text) and re-score. If the budget is exhausted and still below threshold, DISCARD the section (do not append it to `sections`) and append the topic to a new `discarded_topics` list. When `questions` is empty, keep sections as today and note scoring was skipped.

    Record the outcome: append a trailing `## Groundedness\n\n- Kept: N sections\n- Discarded: <topic list or none>` block to the report body, and put a one-line `kept=N discarded=M` summary into `ToolResult.output`. Preserve the existing HON-03 behavior: `produced == 0` (no kept sections) still returns `ToolResult(success=False, ...)`. A run where every section is discarded counts as `produced == 0` -> failure.
