<objective>
Add `append_verify_entry(memory, root, results, *, timestamp=None)` to `flowstate/journal.py` — a lightweight pure-Python sibling of `append_run_entry` that records a standalone `flowstate verify` run as one `MemoryKind.RUN` memory entry (tagged `["verify"]`) plus one `.planning/RUNLOG.md` line. This is half of the loop closure (VER-02): the journal half that fires on every verify run.

Purpose: Make verify runs durable context. Because `append_run_entry` requires pipeline step/manifest state and computes artifact deltas, verify needs its own thin writer with no pipeline-state dependency. Verify RUN entries then auto-surface in `flowstate journal` and the `## Since Last Run` prefix layer (which queries `get_by_kind(RUN)` without tag filtering — no layer change needed).
Output: `flowstate/journal.py` (extended), `tests/test_journal.py` (extended).
</objective>

| Boundary | Description |
|----------|-------------|
| results list → append_verify_entry | In-process VerifyResult objects (trusted, FlowState-produced) |
| RUNLOG.md / memory.db writes | Local filesystem + SQLite write surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-06 | Denial of Service | memory.add / SQLite write failing mid-verify | mitigate | memory.add wrapped in try/except: return — verify must never traceback on a journal write (success criterion #4) |
| T-08-07 | Denial of Service | RUNLOG.md unwritable (perms / missing parent) | mitigate | _append_verify_runlog wraps mkdir+open+write in except Exception: pass |
| T-08-08 | Repudiation | verify run leaves no durable record | mitigate | Journal entry is written on EVERY verify run (pass or fail), giving an append-only audit trail |
| T-08-09 | Tampering | failed_signatures injecting markup into RUNLOG | accept | Gate strings are FlowState-authored fixture text, not external untrusted input; plain-text append only |
</threat_model>
