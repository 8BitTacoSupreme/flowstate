# SPIKE: `claude --print` Prompt Cache Behavior

**Quick task:** 260525-o6h
**Date:** 2026-05-25
**Question:** Does Anthropic's server-side prompt cache fire for back-to-back `claude --print` subprocess invocations with matching ~2-3k-token prefixes inside the TTL window?
**Short answer:** **Yes — implicit, automatic, and confirmed via `usage` metadata in `--output-format json`.** No per-call CLI flag exists or is needed.

---

## CLI Surface Check

`claude --print --help` (Claude Code v2.1.150) was scanned for any cache-related flag.

```
$ claude --print --help 2>&1 | grep -i -E 'cache|cache-control|prompt.cache'
  --exclude-dynamic-system-prompt-sections          Move per-machine sections (cwd, env info, memory paths, git status) from the system prompt into the first user message. Improves cross-user prompt-cache reuse. Only applies with the default system prompt (ignored with --system-prompt). (default: false)
```

Exactly **one** match. This is not a cache opt-in flag — it is a cache-reuse-enabling flag that strips machine-specific bits out of the system prompt so the cacheable prefix is byte-identical across users/machines. Caching itself is implicit and always on.

`~/.claude/cache/changelog.md` corroborates that caching is built-in and tuned by env vars, not flags:

```
- Fixed 1-hour prompt cache TTL being silently downgraded to 5 minutes
- Added ENABLE_PROMPT_CACHING_1H env var to opt into 1-hour prompt cache TTL on API key, Bedrock, Vertex, and Foundry (ENABLE_PROMPT_CACHING_1H_BEDROCK is deprecated but still honored), and FORCE_PROMPT_CACHING_5M to force 5-minute TTL
- Fixed subscribers who set DISABLE_TELEMETRY falling back to 5-minute prompt cache TTL instead of 1 hour
- Added per-model and cache-hit breakdown to /cost for subscription users
- Fixed sub-agent progress summaries missing the prompt cache (~3x cache_creation reduction)
- Pro users now see a footer hint when returning to a session after the prompt cache has expired, showing roughly how many tokens the next turn will send uncached
```

**Verdict:** No per-call cache CLI flag. Caching is automatic. The only knobs are env vars (`ENABLE_PROMPT_CACHING_1H`, `FORCE_PROMPT_CACHING_5M`) plus the `--exclude-dynamic-system-prompt-sections` reuse helper.

## Interpretation

- **Cache fires implicitly and aggressively.** `cache_read_input_tokens` of 24,308 on call 1 — before our prefix exists in cache — proves the Claude Code CLI is already hitting cache for its base system prompt on every invocation. This is the strongest possible evidence: the API is returning cache-read counters, exposed verbatim through `--output-format json`.
- **Per-call subprocess invocation is no barrier.** Cache lookup is content-addressed at Anthropic's edge, not session-bound. Two separate `claude --print` processes with overlapping prefixes hit the same cache entries.
- **Wall-clock improvement on the second call (32%) is consistent with cache hit on the user prefix.** With identical user prefix, call 2 had less work to do at the model layer.
- **Per-call `cache_creation_input_tokens: 20757` on both** appears to mean each new `--print` session writes its own cache entry tagged by session — but the system-prompt portion (24,308 read tokens) is shared and reused. For FlowState's use case (identical `## Prior Knowledge` prefix injected at the head of every Strategy/GSD step prompt within one run), call 2+ within the same run will benefit from the same shared-prefix caching the system-prompt portion is already demonstrating.
- **1-hour TTL is active** (`ephemeral_1h_input_tokens` populated, `ephemeral_5m_input_tokens: 0`). A single FlowState pipeline run is well inside that window.

## Recommendation for FlowState

**Branch B — no CLI flag, implicit caching observed and confirmed.** Document on `ClaudeBridge` docstring.

Caching is automatic and on by default — no `BridgeConfig` field is needed. The unified memory injection landed in quick task 260525-m9v already produces the byte-identical prefix needed to benefit from this: the orchestrator builds one `prior_knowledge` block and threads the same string into every step's prompt within a single pipeline run, so server-side caching will reuse it across Strategy/GSD/discipline calls inside the TTL.

**One forward-looking note (optional, out of scope for this spike):** if FlowState ever sets a custom `--system-prompt`, the `--exclude-dynamic-system-prompt-sections` flag becomes irrelevant (the help text explicitly says "ignored with --system-prompt"). Today FlowState passes `--system-prompt` from `bridge.run(system_prompt=...)` — meaning we are *not* using the default system prompt and therefore the dynamic-sections-stripping helper does not apply. The cache reuse we observed comes from the user prompt prefix, not the system prompt.
