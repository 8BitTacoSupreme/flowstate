**Step A — CLI surface check (no API calls):**
1. Run `claude --print --help 2>&1` and capture the full output to a scratch buffer.
2. Grep for cache-related flags: `claude --print --help 2>&1 | grep -i -E 'cache|cache-control|prompt.cache'`.
3. List `~/.claude/` and skim any docs/readme: `ls -la ~/.claude/ && find ~/.claude -maxdepth 3 -name '*.md' 2>/dev/null | head -20`. Grep matches for "cache" in any docs found.
4. Record verdict: "CLI exposes cache flag X" OR "CLI exposes no cache flag — only implicit server-side caching is testable".

**Step B — Build a deterministic ~2-3k-token prefix:**
- Use a public-domain block: concatenate the opening of *Pride and Prejudice* (Project Gutenberg) repeated until ≥ ~12,000 characters (rule of thumb: ~4 chars/token → ~3k tokens, comfortably above the 1024-token minimum).
- Alternative if curl is undesirable: hardcode a long string inline in a tiny shell snippet. The actual content does not matter — only that it is byte-identical between both calls and ≥ 1024 tokens.
- Append a 1-line varying suffix per call so the model has something to answer, e.g. `\n\nIn one short sentence, what is the protagonist's surname? (call 1)` and `(call 2)`.

**Step C — Two back-to-back calls inside the 5-min TTL:**
1. Build the prefix once into a shell variable `PREFIX`.
2. Call 1: `time claude --print --model haiku --output-format json -- "$PREFIX\n\nIn one short sentence, what is the protagonist's surname? (call 1)" > /tmp/o6h-call1.json 2> /tmp/o6h-call1.time`
3. Call 2 (immediately, same shell): `time claude --print --model haiku --output-format json -- "$PREFIX\n\nIn one short sentence, what is the protagonist's surname? (call 2)" > /tmp/o6h-call2.json 2> /tmp/o6h-call2.time`
4. If `--output-format json` is not supported on this CLI version (verified in Step A) → fall back to `text` and capture only wall-clock time. Note the limitation explicitly in SPIKE.md.
5. Inspect both JSON outputs for any usage metadata: `jq '.' /tmp/o6h-call1.json` and `jq '.' /tmp/o6h-call2.json`. Look for keys like `cache_read_input_tokens`, `cache_creation_input_tokens`, or anything under a `usage` object.
6. If a call fails (auth, network, model not available), record the exact error in SPIKE.md and try once with `--model sonnet`. If both fail, document the failure and stop — SPIKE.md still gets written with the CLI surface finding and the "could not empirically confirm" conclusion, and Task 2 proceeds based on the surface check alone.
