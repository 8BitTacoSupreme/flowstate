# Phase 260709-d64 Plan 01: Extend bench/longmemeval_qa.py — Reader Path + OpenAI Canary Summary

LongMemEval reader path upgraded with a LME-tuned instruction constant, optional OpenAI reader routing, and an upfront canary guard that hard-stops on API failure before the scoring loop runs.

## What Was Built

**bench/longmemeval_qa.py** — three targeted changes:

1. `_READER_INSTRUCTION` module-level constant: prior-session framing template with `{question_date}` placeholder. Formatted per-instance and passed to `_g._answer(..., instruction=...)` via the existing seam — replaces grounding's generic "Answer concisely and specifically."

2. `_answer_one` signature extended with `*, provider="claude"`. Claude path passes the tuned instruction via `instruction=` kwarg. OpenAI path routes through `_openai_chat` with instruction embedded in the user message. Default behavior byte-identical for all existing callers.

3. `_run_qa` generalized: `use_openai = judge_provider=="openai" or reader_provider=="openai"`, prereq check fires for either, upfront canary probes each unique openai model before the scoring loop (deduped set). None canary → prints "openai canary failed for model '<m>'" + return 1. Reader auto-upgrade: `reader_model=sonnet + reader_provider=openai → gpt-4-turbo`. Output JSON gains `reader_provider` and effective `reader_model`. CLI gains `--reader-provider`.

**tests/test_longmemeval_qa.py** — 7 new offline tests + `reader_provider` param in `_make_args`.
