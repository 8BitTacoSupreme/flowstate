<objective>
Extend `MemoryStore` (flowstate/memory.py) with the backing vector store: load sqlite-vec on the
existing `_conn`, create a `memories_vec` vec0 virtual table keyed to the memory `rowid`, embed on
`add`/`update`/`add_many`, and lazily backfill missing vectors on open. Every vector path is guarded
by `embedder.available()` — when the embedder is absent the store behaves exactly as today (FTS5
only), never blocking startup and never raising.

This is the vector half of the Phase 9 foundation. It does NOT add semantic retrieval to
`get_context()` (that is Phase 10) — only write + backfill + the table. A private KNN helper seam may
be established if it keeps the code clean, but no retrieval behavior changes here.

Purpose: additive `memory.db` change (new vec0 table only). The `flowstate.json` state schema and its
migration ladder are UNCHANGED — do not touch state.py.

Output: extended `flowstate/memory.py` and `tests/test_memory.py` additions (no model/network).
</objective>

| embedder absence → FTS5 fallback | When the embedder is unavailable the vec write/backfill paths are skipped entirely; today's FTS5 behavior is the safe default |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

Call `enable_load_extension(False)` immediately after `sqlite_vec.load` to re-scope the load surface — no arbitrary-extension window stays open. Verified by acceptance criterion. |

| T-09-05 | Denial of Service | backfill blocking startup on large stores | accept | Corpus is small (memories table); backfill is lazy/best-effort and never-raises. Performance tuning is explicitly out of scope for v0.6 (REQUIREMENTS.md). |
| T-09-SC | Tampering | npm/pip installs | n/a | No new package installs in this plan; sqlite-vec already pinned in core deps. fastembed install is covered by Plan 09-01's threat model. |
