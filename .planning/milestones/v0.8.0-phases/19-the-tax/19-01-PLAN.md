<objective>
Make `ClaudeBridge.run()` capture the real token/latency usage the `claude --print --output-format json` path already returns but currently discards. `BridgeResult` gains a `usage` field and a `duration_s` field; text-mode `.output` stays byte-identical so no existing caller regresses. The bridge instance also accumulates per-run cumulative totals so downstream plumbing (Plan 02) can read one number per pipeline run.

Purpose: This is the accounting seam the harness has been missing since `bench/` began — the data is one flag away and is currently thrown out (bridge.py:281 returns `output=result.stdout` unparsed even in json mode).
Output: `BridgeUsage` dataclass, `BridgeResult.usage`/`.duration_s`, `ClaudeBridge` cumulative totals, full test coverage.
</objective>

  <name>Task 1: Add BridgeUsage + usage/duration_s to BridgeResult and parse the json path</name>
  <read_first>
    - flowstate/bridge.py — BridgeResult dataclass (~104-109), run() body (~191-298),
      especially the dry_run early return (~211-215), the `output_format == "json"` flag add
      (~230-231), and the success return that sets `output=result.stdout` (~279-284).
    - .planning/quick/260525-o6h-.../260525-o6h-SPIKE.md § Usage Metadata for the exact key names.
  </read_first>
  <behavior>
    - Text mode (default): `.output` equals `result.stdout` exactly as today; `usage` is None.
    - Json mode + well-formed stdout: `.output` equals the parsed top-level `result` string
      (byte-identical to what text mode returns for the same response); `usage` is a
      BridgeUsage(tokens_in, tokens_out, cache_read) from usage.input_tokens/output_tokens/
      cache_read_input_tokens (missing sub-keys default to 0).
    - Json mode + malformed/empty stdout, or no top-level `result` key: never raise; `usage`
      is None and `.output` falls back to raw `result.stdout`; `success` still tracks returncode.
    - `duration_s` is set on every non-dry, non-error return (monotonic clock around the
      subprocess call), regardless of output_format.
  </behavior>
  <action>
    Add a frozen `BridgeUsage` dataclass (fields `tokens_in: int`, `tokens_out: int`,
    `cache_read: int`, all defaulting to 0) near BridgeResult. Add two fields to BridgeResult:
    `usage: BridgeUsage | None = None` and `duration_s: float | None = None` (append after
    existing fields to preserve positional construction). In `run()`, wrap the
    `subprocess.run(...)` call with `time.monotonic()` start/end and compute `duration_s`.
    On the success return, when `output_format == "json"`, parse `result.stdout` with
    `json.loads` inside a try/except: on success extract the top-level `result` key as `.output`
    and build BridgeUsage from the `usage` sub-object (`.get(key, 0)` for each of input_tokens,
    output_tokens, cache_read_input_tokens); on any exception or a missing `result` key, set
    `usage=None` and keep `.output = result.stdout` (the current behavior). Text mode
    (default) path is unchanged: `.output = result.stdout`, `usage=None`. Do NOT change the
    dry_run early return except to leave `usage=None`/`duration_s=None` (dry runs measure no
    real LLM work — document that choice in a one-line comment). Import `time` and `json` at
    module top. Never raise from parsing — honor the existing never-raise/`success` contract.
  </action>
  <acceptance_criteria>
    - `BridgeUsage` exists with fields tokens_in/tokens_out/cache_read; `BridgeResult` has
      `usage: BridgeUsage | None` and `duration_s: float | None`.
    - New test in tests/test_bridge.py asserts text-mode `.output` is byte-identical to the raw

    - New test asserts json-mode `.output` equals the stubbed `result` string and

    - New test asserts json-mode with malformed stdout (or absent `result` key) returns

    - `uv run python -m pytest tests/test_bridge.py -x` passes; `uv run ruff check flowstate/bridge.py` clean.
  </acceptance_criteria>
